iPaper Flipbook · Accessibility

WCAG 2.1 — Development Backlog

Concrete initiatives needed to reach WCAG 2.1 AA, from a checklist review with the team. Items already in place are listed for reference.

Area: V Viewer · E Editor · P PDF-processing Priority = fix order, not necessity. P0 & P1 are both required for AA; P2 is beyond baseline. P0 = severe / fix first · P1 = required, lower severity · P2 = not a criterion.

Does completing this list make iPaper compliant?

No — not automatically. It makes iPaper capable of AA conformance, pending audit:

  1. All-or-nothing: all ~50 A/AA criteria must pass. These items close the known gaps, not the rest.
  2. "Done" ≠ verified until tested with screen readers, keyboard-only and zoom.
  3. Per-catalog & client-dependent: alt text, captions, language and PDF quality come from the client.
  4. Not every criterion is listed (e.g. 1.4.1 colour, 1.3.3 sensory, full contrast, 2.5.x pointer).

A conformance claim is only defensible after these ship, the “already implemented” items are verified, and a full audit on representative catalogs (incl. a scanned PDF and a media/hotspot-heavy one) passes. That audit produces the VPAT.

On priority & compliance: every P0 and P1 item below maps to a Level A or AA success criterion, so all of them are required for AA — two “P1” items (#5, #6) are even Level A. Priority is fix-order/severity, not necessity. Only P2 (#10) is optional for conformance.

Items requiring development

1

Alt text for image Enrichments

P01.1.1 (A)AreaE + VCxLow–MedClientHigh

Build: an alt-text field (+ “decorative” toggle) on every image Enrichment; render as alt/aria-label, never the filename. Warn when missing.

Why: 1.1.1 (Level A) requires a text alternative for informative images. One un-alt’d image Enrichment fails Level A for the whole catalog. Confirmed gap today.

2

Captions for MP4 video Enrichments

P0 · confirmed gap1.2.2 (A) · 1.2.3 (A) · 1.2.5 (AA)AreaE + VCxMediumClientHigh
  • YouTube / Vimeo: captions and accessible controls are handled by those players — only client guidance needed.
  • MP4 (the gap): iPaper does not accept caption uploads today, so MP4 with speech can’t be captioned. Add .vtt caption-track upload via <track> + an optional transcript field + a “no captions” editor warning.

Why: 1.2.2 (Level A) requires captions for spoken-audio video. Clients create captions; iPaper must provide the mechanism — for MP4 it’s missing, making iPaper the blocker. Audio description: transcript covers 1.2.3 (A); 1.2.5 (AA) only applies to visual-only content and needs no AD tool — clients supply an alternate version.

3

Autoplay audio & media controls

✓ Already handled1.4.2 (A)AreaV
  • Audio is never auto-played, so 1.4.2’s trigger never fires.
  • Native browser MP4 controls are keyboard- and screen-reader-accessible.

No work needed. Captions for spoken-audio video are tracked separately under item 2.

4

PDF reading-order / tag detection

P0–P11.1.1 · 1.3.2 (A)AreaP + ECxHighClientMed

Build: use the PDF tag tree where present; detect columns/headings to repair order for untagged PDFs; flag scanned/untagged uploads with a quality warning. (Text is already exposed in order for clean PDFs — this hardens the messy ones.)

Why: 1.3.2 (Level A) needs the correct reading sequence to be programmatic. For a rasterized flipbook the text layer is the screen-reader experience; a scrambled multi-column/untagged PDF reads as nonsense.

5

Modal focus trap & return

◐ Partially implementedP12.4.7 (AA) · 2.4.3 · 2.1.2 (A)AreaVCxMedium
  • Tab order through all elements (incl. Enrichments); standard blue focus ring; modals close on Esc; ARIA labels present. Whole-page focus reads the extracted PDF text.
  • Trap focus inside open modals/overlays (Tab must not reach the page behind).
  • Return focus to the trigger on close.

Why: 2.4.3 (A) needs focus to stay meaningful — a modal that doesn’t trap/return focus strands keyboard users (also 2.1.2). The indicator and Esc parts are done; only the modal lifecycle remains.

6

Skip link & ARIA landmarks

✗ Confirmed missingP12.4.1 · 1.3.1 (A)AreaVCxMedium

Build: a “skip to content” link as the first focusable element + landmark regions (banner/main/navigation). No skip option exists today.

Why: 2.4.1 (Level A) requires a way to bypass repeated chrome; without it keyboard/SR users tab through the toolbar on every page. (Publitas marked this N/A — that wouldn’t survive an audit.)

7

aria-live status messages

P14.1.3 (AA)AreaVCxLow–Med

Build: aria-live regions for add-to-cart, loading and search-result changes that don’t move focus.

Why: 4.1.3 (AA) — without it, “Added to cart” / “12 results” is silent for screen-reader users.

8

Language of parts

P13.1.2 (AA)AreaE + VCxMediumClientHigh

Build: let clients mark sections in a different language (per-section lang), beyond the page language already set via Region Settings.

Why: 3.1.2 (AA) — a French passage read by an English TTS voice is unintelligible. Applies to multilingual catalogs.

9

Reflow / zoom validation

P11.4.4 · 1.4.10 (AA)AreaVCxHighClientMed

Build: test/fix at 200% zoom and 400%/320px reflow; ensure reader zoom + text/PDF fallbacks are easy escape hatches. Document honestly.

Why: 1.4.4/1.4.10 (AA) — fixed-size page images risk forcing horizontal scroll at high zoom. Hardest item to pass cleanly.

10

In-editor accessibility checker

P2 · differentiatorAuthoring: 1.1.1 · 1.2.2 · 1.4.3 · 2.4.4 · 3.1.2AreaECxHigh

Build: a panel flagging missing alt text, captions, accessible names, low contrast, missing language and untagged PDFs — with fixes and an optional publish gate.

Why: not a criterion itself, but it’s how a UGC platform reaches conformance in practice — and the strongest basis for a public accessibility claim.

Summary

  • P0 (blockers): #1 alt text · #2 MP4 captions · #4 PDF reading-order.
  • P1: #5 modal focus (partial) · #6 skip link (missing) · #7 aria-live · #8 language of parts · #9 reflow/zoom.
  • P2: #10 editor accessibility checker (biggest differentiator).
  • Already covered: #3 autoplay/controls; YouTube/Vimeo captions; plus the verified-pending items above.
  • Heaviest lifts: #4 PDF reading-order, #9 reflow/zoom, #10 editor checker.
  • Client-dependent: alt text (#1), captions (#2), language (#8), PDF quality (#4) — platform provides the field & warns.

Opportunity: “PDF ripper” → an optional “Text version” page layer

Beyond compliance — one feature that pays off for AI search (GEO), users and accessibility at once.

Today: extracted page text lives inside window.staticSettings.pageTexts (a <script>). It’s present, but many AI crawlers and SEO tools ignore script contents — so a customer’s “AI can’t read our flipbook” claim looks true from the outside.

Idea: an improved extraction module (internal: PDF ripper) feeds an optional viewer feature that renders each page’s text as semantic HTML in the DOM — e.g. a collapsible “Read page text” panel (<details> with a heading + paragraphs per page).

  • Crawlers / AI (GEO): page text becomes crawlable and quotable in AI answers, not buried in JS.
  • Users: copy/paste, in-page search (Ctrl-F), and a readable fallback.
  • Accessibility: clean reading-order text for screen readers — reinforces items 1 & 4.

Framing: ship it as a genuine, visible/collapsible text layer — not hidden SEO text. That removes the original cloaking concern behind the JPEG-only design; Google treats legitimate accordion / “text version” UX as fine. Pair with crawlable per-page <a href> links, per-page metadata (title/description/canonical) and a crawlable PDF download. Quality depends on the ripper’s extraction & reading-order (item 4); keep the feature opt-in.
