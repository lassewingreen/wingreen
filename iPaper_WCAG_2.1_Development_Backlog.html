WCAG 2.1 — Development Backlog

Concrete initiatives needed to reach WCAG 2.1 AA, from a checklist review with the team. Items already in place are listed for reference.

Area: V Viewer · E Editor · P PDF-processing Priority = fix order, not necessity. P0 & P1 are both required for AA; P2 is beyond baseline. P0 = severe / fix first · P1 = required, lower severity · P2 = not a criterion.
Bottom line
  • Not yet WCAG 2.1 AA compliant, but close — most of the work is done or scoped.
  • 3 blockers (P0): alt text for image Enrichments, MP4 captions, PDF reading-order (PDF ripper).
  • 4 further required (P1): skip link, aria-live, language of parts, reflow/zoom.
  • A compliance claim requires these shipped, current features verified, and an external audit (→ VPAT).
  • One optional growth bet, for your call: a pre-rendered “Text version” for AI/GEO visibility — same PDF ripper work as the blocker above.

Does completing this list make iPaper compliant?

No — not automatically. It makes iPaper capable of AA conformance, pending audit:

  1. All-or-nothing: all ~50 A/AA criteria must pass. These items close the known gaps, not the rest.
  2. "Done" ≠ verified until tested with screen readers, keyboard-only and zoom.
  3. Per-catalog & client-dependent: alt text, captions, language and PDF quality come from the client.
  4. Not every criterion is listed (e.g. 1.4.1 colour, 1.3.3 sensory, full contrast, 2.5.x pointer).

A conformance claim is only defensible after these ship, the “already implemented” items are verified, and a full audit on representative catalogs (incl. a scanned PDF and a media/hotspot-heavy one) passes. That audit produces the VPAT.

On priority & compliance: every P0 and P1 item below maps to a Level A or AA success criterion, so all of them are required for AA — one “P1” item (#6 skip link) is even Level A. Priority is fix-order/severity, not necessity. Only P2 items (#5 polish, #10 checker) are optional for conformance.

Items requiring development

1

Alt text for image Enrichments

P01.1.1 (A)AreaE + VCxLow–MedClientHigh

Build: an alt-text field (+ “Image is decorative” toggle) on every image Enrichment; render as alt/aria-label, never the filename. Warn when missing.

Why: 1.1.1 (Level A) requires a text alternative for informative images. One un-alt’d image Enrichment fails Level A for the whole catalog. Confirmed gap today.

2

Captions for MP4 video Enrichments

P0 · confirmed gap1.2.2 (A) · 1.2.3 (A) · 1.2.5 (AA)AreaE + VCxMediumClientHigh
  • YouTube / Vimeo: captions and accessible controls are handled by those players — only client guidance needed.
  • MP4 (the gap): iPaper does not accept caption uploads today, so MP4 with speech can’t be captioned. Add .vtt caption-track upload via <track> + an optional transcript field + a “no captions” editor warning.

Why: 1.2.2 (Level A) requires captions for spoken-audio video. Clients create captions; iPaper must provide the mechanism — for MP4 it’s missing, making iPaper the blocker. Audio description: transcript covers 1.2.3 (A); 1.2.5 (AA) only applies to visual-only content and needs no AD tool — clients supply an alternate version.

3

Autoplay audio & media controls

✓ Already handled1.4.2 (A)AreaV
  • Audio is never auto-played, so 1.4.2’s trigger never fires.
  • Native browser MP4 controls are keyboard- and screen-reader-accessible.

No work needed. Captions for spoken-audio video are tracked separately under item 2.

4

PDF reading-order & structure — updated PDF ripper

P0–P11.1.1 · 1.3.2 (A)AreaP + ECxHighClientMed
  • Today: on page focus the screen reader reads the page aloud — each page is a focusable <section tabindex="0"> with an <h2>Page number N</h2> + the extracted text. The text is exposed.
  • But it’s one flat run-on <p> — headings, prices, refs and sizes mashed together with no reading order (multi-column catalogs scramble badly). Needs an updated PDF ripper: column/heading detection, logical reading order, segmented structure (headings vs paragraphs), and a quality flag for scanned/untagged PDFs.

Why: 1.3.2 (Level A) needs the correct reading sequence, not just presence. The text layer is the screen-reader (and crawler) experience; a jumbled dump reads as nonsense even though text is technically there. Same ripper powers the optional GEO “Text version” feature mentioned below.

5

Modal focus behaviour

✓ Meets WCAG 2.1 AAP2 · polish only2.4.7 (AA) · 2.4.3 · 2.1.2 (A)AreaV
  • Tab order through all elements; standard blue focus ring (2.4.7); modals close on Esc; ARIA labels; whole-page focus reads the extracted PDF text.
  • VoiceOver-tested: Product View & Cart modals are escapable — no keyboard trap (2.1.2 ✓) — and focus returns to the catalog (logical order, 2.4.3 ✓).
  • Optional polish (not required): return focus to the exact triggering CTA, and/or trap focus within the modal. Both are ARIA APG best practices, not WCAG 2.1 success criteria.

Why it’s fine: WCAG 2.1 requires no keyboard trap, logical focus order and visible focus — all met. Returning to the exact CTA is an ARIA best practice, not a 2.1 requirement. (Not trapping focus is fine for 2.1; only WCAG 2.2 tightens this.)

6

Skip link & ARIA landmarks

P12.4.1 · 1.3.1 (A)AreaVCxMedium

Build: a Flipbook viewer “skip to content” link as the first link (top-left), appearing whenever a user tabs in or a screen reader is active; + landmark regions (banner/main/navigation). No skip option exists today.

Why: 2.4.1 (Level A) requires a way to bypass repeated chrome; without it keyboard/SR users tab through the toolbar on every page.

7

aria-live status messages

P14.1.3 (AA)AreaVCxLow–Med

Build: aria-live regions for add-to-cart, loading and search-result changes that don’t move focus.

Why: 4.1.3 (AA) — without it, “Added to cart” / “12 results” is silent for screen-reader users.

8

Language of parts

P13.1.2 (AA)AreaE + VCxMediumClientHigh

Build: let clients mark sections in a different language (per-section lang), beyond the page language already set via Region Settings.

Why: 3.1.2 (AA) — a French passage read by an English TTS voice is unintelligible. Applies to multilingual catalogs.

9

Reflow / zoom validation

P11.4.4 · 1.4.10 (AA)AreaVCxHighClientMed

Build: test/fix at 200% zoom and 400%/320px reflow; ensure reader zoom + text/PDF fallbacks are easy escape hatches. Document honestly.

Why: 1.4.4/1.4.10 (AA) — fixed-size page images risk forcing horizontal scroll at high zoom. Hardest item to pass cleanly.

10

In-editor accessibility checker

P2 · differentiatorAuthoring: 1.1.1 · 1.2.2 · 1.4.3 · 2.4.4 · 3.1.2AreaECxHigh

Build: a panel flagging missing alt text, captions, accessible names, low contrast, missing language and untagged PDFs — with fixes and an optional publish gate.

Why: not a criterion itself, but it’s how a UGC platform reaches conformance in practice — and the strongest basis for a public accessibility claim.

Summary

  • P0 (blockers): #1 alt text · #2 MP4 captions · #4 PDF reading-order.
  • P1: #6 skip link · #7 aria-live · #8 language of parts · #9 reflow/zoom.
  • P2 (optional): #5 modal focus polish (exact-CTA return) · #10 editor accessibility checker (biggest differentiator).
  • Already covered: #3 autoplay/controls; #5 modal focus (meets 2.1 AA — VoiceOver-tested); YouTube/Vimeo captions; plus the verified-pending items above.
  • Heaviest lifts: #4 PDF reading-order, #9 reflow/zoom, #10 editor checker.
  • Client-dependent: alt text (#1), captions (#2), language (#8), PDF quality (#4) — platform provides the field & warns.

Opportunity: “PDF ripper” → an optional pre-rendered “Text version”

Exploratory — pending decision & further research. Optional; relevant to AI/GEO visibility, users and accessibility.

Today (three versions of the page): (1) raw HTML source holds the text in window.staticSettings.pageTexts (a <script>); (2) after the viewer JS runs, the rendered DOM gets the per-page accessibility <section tabindex="0"> with the page text — great for screen readers; (3) what an AI crawler fetches often misses both, because many crawlers don’t run the viewer JS, snapshot before that section is injected, or filter accessibility-only/duplicate content. So the text exists — it’s a rendering/timing problem, not a “text doesn’t exist” one. (Googlebot is more capable and won’t penalise legitimate screen-reader text; most AI crawlers aren’t Googlebot.)

Lever: the updated PDF ripper (item 4) feeds an optional, server-side / pre-rendered “Text version” — each page’s structured text placed in the initial HTML response per page URL, not only after the app initialises. Ideally a visible/collapsible “Read page text” panel (<details> with heading + paragraphs).

  • Crawlers / AI (GEO): text is captured even by crawlers that don’t execute JS.
  • Users: copy/paste, in-page search (Ctrl-F), and a readable fallback.
  • Accessibility: clean reading-order text for screen readers — reinforces items 1 & 4.

Framing: ship it as a genuine, visible/collapsible text layer — not hidden/accessibility-only text — so crawlers treat it as primary, user-facing content. Pair with crawlable per-page <a href> links, per-page metadata (title/description/canonical) and a crawlable PDF download. Quality depends on the ripper’s reading-order (item 4); keep the feature opt-in.
